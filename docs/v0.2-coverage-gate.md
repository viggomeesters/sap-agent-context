# SAP Agent Context v0.2 coverage gate

`schema/completeness-matrix.yaml` remains the starter safety gate used by
`make check`: it proves the repository can build and serve representative starter
bundles without schema, freshness, or evaluation regressions.

`schema/sap-agent-context-v0.2-coverage.yaml` is stricter. It is the v0.2
readiness gate for turning this repository into a genuinely useful public clone
for AI agents. The current repository is expected to pass this gate; future work
should raise thresholds or add domain checks only when the added evidence can pass
mechanically.

## Current status

The v0.2 fill campaign has closed the original blocking gaps. The gate is now a
fail-hard regression check, not a report-only reminder.

Current coverage includes:

- material/product and business-partner source-backed domain coverage;
- migration cockpit/template-label coverage;
- exact-page/catalog-entry source specificity at the v0.2 threshold;
- release-applicability metadata at the v0.2 threshold;
- representative finance/AP, procurement, sales/output, authorization,
  integration/API, extensibility/analytics and core domain coverage;
- runtime retrieval fixtures that guard against adjacent generic context passing
  as domain-specific evidence.

## How to run

```bash
make audit-v02-gap-report
```

Equivalent direct command:

```bash
uv run sap-agent-context audit-completeness \
  --matrix schema/sap-agent-context-v0.2-coverage.yaml
```

Expected current result: `status: passed` with `findings: []`. Any blocking
finding must produce a non-zero command exit via `make audit-v02-gap-report`.

## Audit scope boundary

Residual risk: audit scope is repo-level, not exhaustive claim-by-claim SAP
content review across all domain-pack YAML. Gates, runtime behavior,
source/access boundaries, CI semantics and the from-zero ontology layer are
covered; full SAP claim curation remains a separate pass.

## What v0.2 checks beyond starter coverage

- higher total item and per-kind minimums;
- concrete SAP field definitions and field maps per domain;
- exact-page/catalog-entry source specificity, not only SAP root links;
- release applicability metadata at scale;
- domain-specific kind counts for master data, migration cockpit, AP,
  procurement, sales/output, authorizations, integrations, and
  extensibility/analytics;
- representative queries that must contain the right item kinds and quality
  dimensions instead of passing via adjacent generic context.

## Non-negotiables

- Do not weaken v0.2 just to make the campaign green.
- Do not copy proprietary SAP documentation; use source URLs and own concise
  claims.
- Do not add customer/client/project data, screenshots, tenant URLs, secrets, or
  internal mappings.
- Keep `freshness.review_after` and `freshness.expires_at` on every knowledge
  item.
- Keep `bundle_kind: sap_fo_context_bundle` until an explicit migration plan
  exists.
